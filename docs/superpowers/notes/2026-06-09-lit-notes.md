## Li & Ning 2023 — Autonomous GIS: the next-generation AI-powered GIS（2741, IJDE）

- **①架构**：单体原型 **LLM-Geo**（GPT-4 为 decision-making core）。五模块：决策(LLM)、数据采集、数据操作、操作日志、历史检索；五自治目标：self-generating / self-organizing / **self-verifying** / self-executing / self-growing。
- **②规划层/输出**：⭐LLM-Geo 把任务分解为相连数据操作的**有向无环图(DAG)**（solution graph / geoprocessing workflow），每操作是 LLM 实现的函数，再合并代码执行。→ **"DAG 用于地理处理工作流"的思想源头**。
- **③错误处理**：提出 self-verifying 目标，但自承原型 "lacking ... logging and code testing"，无结构化"执行错误→规划缺陷"回溯。
- **④评测**：3 个 case study，定性"accurate"；无规模基准、无成功率。
- **⑤引证角色**：**(a) 奠基引用**（Autonomous GIS 概念与五目标源头）+ DAG 思想前驱。
- **⚠️ 影响**：创新点①（DAG 显式依赖）**非首创**，须改为"在 LLM-Geo 的 solution-graph 上增加显式 I/O 类型契约与可验证性"。novelty 落在②③。其 self-verifying 早期目标可作本文③的"未竟之志"承接。

## Lin 2026 — GeoAgent: hierarchical LLM-based multi-agent architecture（11449, IJGIS）

- **①架构**：分层多智能体三层——**Planning Layer**（解析需求、制定执行计划）/ **Execution Layer**（manager agent 调度执行 agent）/ **Review Layer**（自主验证与优化）。集成三套工具集。
- **②规划层/输出**：有独立 Planning Layer，输出 execution plans（具体结构化形式待读 §3.1 细节确证；其卖点是分层协作，未强调"显式依赖 DAG / 语义约束中间层"）。
- **③错误处理**：⭐有 **Review Layer 做"autonomous verification and optimization"**——即它**确实有反馈优化**。⚠️这对本文 1.4"即便引入 Reviewer 也无法回溯规划层"的论断构成挑战：措辞须精确化为"GeoAgent 的 review 是对执行结果/流程的验证优化，而非建立在『显式语义约束+DAG 结构』之上、可把错误精准定位到规划层某缺陷类型的回溯"。**需读其 Review Layer 细节确证**。
- **④评测**：147 tasks；94.56%(GPT-4o)/95.24%(DeepSeek-V3)；消融证明 Planning+Review 层关键。
- **⑤引证角色**：(a) 标杆 + (b) 问题源 + (c) 基线 Baseline 2，三重身份成立。但"问题源"批判要精确：不是"它没反馈"，而是"反馈未建立在可精准回溯规划层缺陷的结构化表示上"。
- **⚠️ 架构高度相似**：GeoAgent 三层(Planning/Execution/Review) ≈ 本文 L2/L3/L4。差异须落在：本文独立的 **L1 知识层** + L2 内部"**语义约束→DAG 两步**" + L4 基于结构化表示的**按缺陷类型精准回溯**。否则易被视为 GeoAgent 的增量。

## Xiao 2026 — GeoContra: From Fluent GIS Code to Verifiable Spatial Analysis with Geography-Grounded Repair（12032, arXiv 2605.00782, 2026-05-01）

> ⚠️⚠️ **正面强竞品 / 并行工作**。与本文核心主张高度重叠，必须正面 engage 并锐化区分。

- **①架构**：不是 agent 规划框架，而是**包裹"LLM 产 Python GIS 代码"的"验证—修复"框架**（geography-grounded verification & repair）。
- **②表示**：⭐把每个任务表示为 **executable geospatial contract** = τ=(q 自然语言, D 数据集+schema+CRS, Y 期望输出, C 空间约束[CRS/拓扑/谓词语义/度量/spatial support], M+ 期望操作, M− 禁止捷径)。**——与本文"空间语义约束表示(七要素)"高度重叠**。
- **③错误处理**：⭐三层验证（static Vs / runtime Vr / task-semantic Vt）+ bounded repair loop；明确主张"用 **geographic violations 而非 generic self-reflection** 来修复"。**——正是本文"结构化回溯 vs Reviewer 后验修补"的同款卖点**。
- **④评测**：7,079 真实任务 / 15 个 Greater Boston 区域 / 9 类任务 / 11 开源模型 600 runs；spatial correctness：DeepSeek-V4 47.6→77.5、Kimi-K2.5 57.7→81.5、开源均 +26.6pp；自称"首个面向 contract-verifiable LLM GIS workflow 的大规模基准"。
- **⑤引证角色**：**最强相关工作，必须正面对比区分**。
- **⚠️ 重叠与差异（本文立足点）**：
  - **重叠**：contract ≈ 语义约束表示；violation-guided repair ≈ 结构化错误回溯；共同主张"from fluent code → verifiable spatial analysis"。
  - **差异（须前置强调）**：(1) GeoContra 面向**代码生成**（wrap 产 Python 代码的 LLM）；本文面向 **agentic 工具调用 / DAG 规划**。(2) GeoContra 的 contract 是**验证用**（生成后检查+repair）；本文的语义约束是**生成用**（规划中间层，驱动 DAG 生成）——前者验证代码、后者构造计划。(3) GeoContra 靠重新 prompt LLM 做**代码 repair**；本文把错误映射回 **DAG 的具体节点/边/约束类型**做结构化**定向重规划**。(4) 规模/领域：7079 Boston 代码生成 vs 90 大湾区 DAG-planning agent。

## Mansourian & Oucheikh 2026 — Bridging NL and GIS: multi-agent framework（11649, IJDE）

- **①架构**：多智能体（任务分解 / 上下文工具检索 / 代码生成 / 迭代调试）+ CoT + RAG；微调 LLaMA 产 PyQGIS 代码。
- **②规划/输出**：任务分解为子任务→PyQGIS 代码；**承认"工具间顺序依赖（上游输出=下游输入）"但未形式化为 DAG/语义**。
- **③错误处理**：迭代 self-refinement / self-debugging → **"moderate gains"（仅适度提升）**。明列现有系统 4 缺口，其中 **#2"缺乏对空间逻辑的实时验证→易错"、#4"把执行当单步代码生成、忽视迭代反馈本质"**——直接支撑本文动机。
- **④评测**：1–2 工具 ≤100% 执行成功 / 87.5% 语义正确；复杂多步显著下降。
- **⑤引证角色**：**(b) 问题源**（缺实时空间逻辑验证、迭代仅适度增益）+ (a) 多智能体相关。→ 1.4 可据此把"Reviewer 无法回溯"精确化为"现有迭代/审查仅适度增益、未建立结构化回溯"。

## Luo 2026 — GeoJSON agents: function calling vs. code generation（11447, Big Earth Data）

- **①架构**：多智能体（planner 分解子任务 + worker 经 function-calling 或 code-gen）+ 结果整合；产 GeoJSON；迭代 refine。**（与 GeoAgent 同组：Lin/Xu/Wu/Du）**
- **②规划/输出**：planner 系统分解为可执行子任务，但产物是 GeoJSON 操作/代码，**非显式类型化 DAG**。
- **③错误处理**：迭代 refinement；强调通用 LLM 多工具协调时"incoherent steps / poor stability"。
- **④评测**：70 任务（basic/intermediate/advanced）；code-gen 97.14% vs function-calling 85.71% vs 通用 48.57%。
- **⑤引证角色**：**(b) 问题源**（多工具不稳定/步骤不连贯——对应 1.4"顺序错乱、数据流断裂"）+ function-calling vs code-gen 路线对照。

## Akinboyewa 2025 — GIS Copilot: towards an autonomous GIS agent（11456, IJDE）

- **①架构**：LLM 嵌入 QGIS 插件；agent 把任务"拆成步骤、选 GIS 工具、产+执行 Python 代码"，提供工具文档。**（Li & Ning 团队）**
- **②规划/输出**："breaks task into steps and selects tools"——**非正式分解，无显式 DAG/语义约束层**。
- **③错误处理**：未强调结构化回溯；高级任务的完全自主执行仍困难。
- **④评测**：100+ 任务（basic/intermediate/advanced）；basic/intermediate 高成功率，advanced 未达完全自主。
- **⑤引证角色**：**(a) 代表性系统**（1.2 已引），体现"工具调用+代码生成、informal 分步"范式与平台集成（QGIS）思路。

## Zhang & Gao 2025 — GeoAnalystBench: GeoAI Benchmark for Spatial Analysis Workflow（12175, TGIS）

- **基准**：50 个 Python 地理处理任务（真实问题、GIS 专家校验），每任务配 minimum deliverable；评测维度 **workflow validity / structural alignment / semantic similarity / CodeBLEU**。ChatGPT-4o-mini 95% validity；**空间关系检测与选址最难**。
- **⑤引证角色**：⭐**本文第五章评测的直接先例**——其"工作流有效性 / 结构对齐 / 语义相似"≈ 本文过程指标（节点/边匹配）与"DAG vs 参考 DAG 对齐"；可引为评测设计依据与对照（第五章 5.4 / 5.5）。
